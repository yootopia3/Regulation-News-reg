# 리팩토링 시작 템플릿

기존 코드 리팩토링을 시작할 때 이 템플릿을 채워서 AI에게 전달한다.
신규 프로젝트용 1~7단계는 건너뛴다. 이 한 장만으로 SKILL.md 흐름(논의 → task 생성 → `_runner/run-phases.py` 실행)으로 곧장 진입한다.

---

## 1. 목적

{왜 리팩토링하는가. 한두 줄. 예: "초기 프로토타입으로 만들어 구조가 엉망이라 유지보수가 어렵다. 동작은 그대로 유지하면서 코드 품질을 개선한다."}

## 2. 현재 동작 (변하면 안 되는 것)

리팩토링 전후로 동일하게 유지되어야 하는 동작을 명시한다. AC 검증의 기준이 된다.

- {기능 1: 예 — "10분마다 기관별 RSS를 수집해 articles 테이블에 upsert한다"}
- {기능 2}
- {외부 인터페이스(API, CLI, 환경변수, DB 스키마) 중 유지해야 하는 것}

## 3. 건드리지 않을 범위 (out of scope)

이번 리팩토링에서 손대지 않을 것을 명시한다. scope 폭주 방지.

- {예 — "DB 스키마는 변경하지 않는다"}
- {예 — "프론트엔드는 이번 task에서 제외"}
- {예 — "GitHub Actions 워크플로우는 그대로 둔다"}

## 4. 회귀 체크리스트

자동 테스트가 부족할 때 수동으로 확인할 항목. 각 phase의 AC에 일부씩 분배한다.

- [ ] {빌드 성공 — 예: `pip install -r requirements.txt && cd web && npm run build`}
- [ ] {핵심 동작 1 수동 확인}
- [ ] {핵심 동작 2 수동 확인}
- [ ] {외부 호출 결과가 기존과 동일한지 비교}

## 5. 우선순위

리팩토링 항목을 영향도/난이도/의존성 순으로 정렬. 위에서 아래 순서로 phase가 만들어진다.

1. {가장 먼저 손볼 것}
2. {그다음}
3. {그다음}

## 6. AC (전체 task 완료 기준)

이 task가 끝났다고 말할 수 있는 객관적 기준.

- {예 — "build_command가 통과한다"}
- {예 — "회귀 체크리스트 모두 통과"}
- {예 — "기존 동작에 변화 없음"}

---

위 내용이 채워지면 SKILL.md의 흐름으로 진입하라:

1. `spec/`(있다면), 관련 `docs/`, 소스 코드를 읽어 현재 상태를 파악한다.
2. 위 항목 중 모호한 부분, 추가로 결정해야 할 점을 사용자와 논의한다.
3. 합의 후 `prompts/task-create.md`에 따라 task와 phase 파일을 생성한다.
4. `python3 _runner/run-phases.py {task-dir}`로 실행한다.

`tasks/{id}-{name}/index.json`에는 다음 필드를 권장한다:

```json
{
  "commit_prefix": "refactor",
  "branch_prefix": "refactor",
  "push_on_complete": false,
  "build_command": "<프로젝트 빌드 커맨드>"
}
```
